75  /*
76  Construct a pdf_tile_cache.
77 
78  The "requesters" vector contains shared_ptrs to implementations of the pdf_tile_cache_requester
79  interface, which are queried by the paint loop to decide which page/draw_params/clip_rect to work
80  on next.
81 
82  Typically this will contain two requesters: one instance of pdf_tile_cache_page_view_requester,
83  which is owned by a scrolling galley of PDF content and reports which pages are visible and at what
84  scale, and one instance of pdf_tile_cache_thumbnail_requester, which makes sure that thumbnail-sized
85  renderings of all pages in a specified "neighborhood" of pages are available for use by a thumbnail
86  view, or to improve the scrolling experience in the galley.
87 
88  A model of the "pdf tile cache client" (in this example, my_client) concept must provide the following function:
89 
90  my_bitmap make_bitmap( my_client const& client,
91  int point const& size, void const* pixels, size_t scan_line_stride );
92 
93  This function must return an instance of a model of the Bitmap concept, initialized with the specified size,
94  and data pointed to by the pixels parameter. The pixel format is always BGRx. scan_line_stride is the
95  number of _bytes_ between (x,y) and (x,y+1) in the source data. It is _not_ guaranteed to be 4 * get_x(size).
96  There is no requirement that the created image object preserve this format or scan_line_stride value in its
97  own internal representation.
98 
99  Models of the Bitmap concept are typically modeled by a thin RAII wrapper around a platform-specific image
100  or texture class. The Bitmap concept is not complex; models of Bitmap must satisfy the following requirements:
101 
102  - Bitmap must be a regular type (i.e., copyable with regular copy semantics)
103  - The copy must be "fast," i.e., a copy of a bitmap shouldn't involve copying all the
104  pixel data contained therein
105 
106  If your native bitmap data type doesn't support "fast" copying, you can wrap it up and use a shared_ptr
107  to your type to achieve the same effect.
108 
109  The objects returned by calls to make_bitmap are passed back to the pdf_tile_cache_client inside the callable
110  provided to pile_tiles (which see).
111 
112  minimum_tile_size describes the minimum number of pixels on the side of a bitmap; bitmaps will always
113  be between 1x and 2x this size on each side. One exception is the bitmaps corresponding to the edge of
114  the page, where clipping may cause these numbers to be smaller.
115  */
116  pdf_tile_cache( PDFTCClient client,
117  std::shared_ptr<document> doc,
118  vector<shared_ptr<pdf_tile_cache_requester>> requesters,
119  int_coord minimum_tile_size = pdf_detail::default_minimum_tile_size );

129  /*
130  Register a callback with the pdf_tile_cache. Callback should be a callable that
131  takes no parameters and returns void. This callback will be called by the
132  pdf_tile_cache whenever new tile content appears in the cache.
133 
134  This is the mechanism by which a view in the GUI is informed that a view
135  update might be useful. The callback should trigger a call to pile_tiles() which
136  will repopulate the GUI with tiles that are more complete and/or closer to the target
137  resolution.
138 
139  The provided callback will be called on an arbitrary thread. The PDFTCClient is free
140  to call pile_tiles() synchronously or not, but we expect most implementations to
141  simply enqueue some event which causes the UI thread to update the view.
142 
143  The callback will be automatically deregistered when the returned pdf_tile_cache_listener
144  is destroyed.
145  */
146  template<typename Callback>
147  pdf_tile_cache_listener add_listener( page_num page, Callback callback );
148 
149  /*
150  Given a page number, draw parameters and a clip rectangle in device space, enumerate a
151  "pile" of tiles that provides the best rendering that can be built out of what's
152  present in the cache.
153 
154  This "pile" is expressed through an ordered sequence of calls into the provided function object,
155  whose signature should match:
156 
157  void f(bitmap_type const& bitmap, int_rect const& where);
158 
159  f() will be called back with various combinations of bitmap/rectangle pairs such that,
160  if the bitmaps are stretched to fit the specified rectangle and painted at the rectangle's
161  location in device space, the result will be the best possible rendering of the page content
162  within the specified clip rectangle. Tiles will frequently overlap and later calls to f()
163  should obscure earlier calls to f().
164 
165  Invocations of f() will always intersect clip_rect, but they may spill outside those bounds.
166  It is the callee's job to clip any pixels outside this boundary, if necessary.
167  */
168  template<typename F>
169  tile_collection pile_tiles( page_num page, draw_params const& draw_params, int_rect const& clip_rect, F&& f );
170
